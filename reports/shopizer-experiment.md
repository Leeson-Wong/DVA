# 变维拆分方法论验证实验报告

## 一、实验概述

### 目的
验证"变维拆分"方法论在真实项目重构中的价值。通过与无方法论（base）、流程规范（Superpowers）、以及两者组合（dims+superpowers）的对比，评估方法论对代码质量和开发效率的影响。

### 实验设计
- **目标项目**：Shopizer（Java 电商开源项目，commit 6a4a0a65）
- **聚焦模块**：sm-shop（310 个 Java 文件）+ sm-core（335 个 Java 文件）
- **实验模型**：GLM-5.1（API 调用）
- **执行方式**：四个独立环境并行执行，五轮递进需求串行推进

### 四个环境

| 环境 | 方法论 | 核心特征 |
|------|--------|---------|
| base | 无 | 仅给需求，不提供任何方法论指导 |
| dims | 变维拆分 | 提供实体/关系基元、变化速率判据、外化手段选择 |
| superpowers | Superpowers 流程 | 提供六步开发流程（Brainstorming→TDD→Review） |
| dims+super | 两者结合 | 变维拆分决定结构，Superpowers 决定流程 |

### 五轮递进需求

| 轮次 | 需求 | 考察点 |
|------|------|--------|
| R01 | 拆解支付类型 if-else 链 | 支付渠道外化能力 |
| R02 | 添加微信支付 | 新类型加入是否只改一处 |
| R03 | 拆解 ProductPriceUtils（721行） | 定价规则外化能力 |
| R04 | 购物车到订单加风控检查 | 流程步骤变更的影响范围 |
| R05 | 新增运费计算规则 | 是否与 R01 架构冲突 |

## 二、关键发现

### 发现 1：方法论拉开差距的拐点在第三轮

Round 01-02（支付分发 + 微信支付），四个环境的差异有限。所有方法论组都选择了 Strategy 模式，base 也能完成任务。

Round 03（定价拆分）开始出现显著分化：
- base 拆分最少（2 维度），5 分钟完成，但无测试
- superpowers 测试最多（68 个），但耗时 30 分钟
- **dims+super 拆分最彻底（PPU 118 行，-83.6%），12 分钟完成，42 个测试**

结论：需求复杂度越高，方法论叠加的价值越明显。

### 发现 2：变维分析的核心价值不只是"怎么拆"，也包括"不需要拆"

Round 05（运费规则）是最有说服力的案例。Shopizer 已有 `ShippingQuoteModule` SPI：

| 环境 | 方案 | 文件数 | 判断依据 |
|------|------|--------|---------|
| base | 直接用 SPI | 2 | 无分析，直觉选择 |
| dims | 策略+适配器桥接 | 5 | 分析了，但过度外化 |
| superpowers | 自建 SPI | 6 | 无分析，从头设计 |
| **dims+super** | **直接用 SPI** | **3** | **变维分析判断"已有外化足够"** |

dims+super 的变维分析识别出：运费规则的变化速率已经被 sm-core 的 `ShippingQuoteModule` 接口正确外化，不需要新的外化点。这避免了 dims 单独使用时的过度设计和 superpowers 的重复建设。

### 发现 3：累积变维分析表是跨轮次的资产

dims 和 dims+super 环境的变维分析表从 R01 累积到 R05，覆盖了：
- 6-12 个实体的状态和变化速率
- 8-11 条关系的变化速率、趋势、策略
- 8-9 个外化决策及其创建轮次和修改记录

这提供了一种独特的代码考古视角——每一轮新增了什么变化维度，之前的决策是否经得起后续需求的考验。

### 发现 4：效率与质量的权衡曲线

| 环境 | 平均耗时 | 平均质量 | 特征 |
|------|---------|---------|------|
| base | 最快 | 最低 | 简单需求够用，复杂需求走弯路 |
| dims | 中等 | 中等 | 分析到位但缺乏测试保障 |
| superpowers | 中等 | 高（测试多） | 流程保障质量，但缺乏结构指导 |
| dims+super | 较慢 | **最高** | 累积资产最多，R03 起弯道超车 |

## 四、结论

### 变维拆分方法论的价值

1. **结构化思考**：用"变化速率是否对齐"替代直觉判断，减少过度设计和设计不足
2. **决策可追溯**：变维分析表记录了每一轮的决策依据，支持跨轮次回溯
3. **克制外化**：方法论的禁忌清单（YAGNI、不要用大炮打蚊子）有效防止了过度抽象
4. **与流程互补**：变维拆分决定"怎么拆"，Superpowers 决定"按什么顺序拆"，两者叠加效果最佳

### 局限性

1. **简单需求下开销不划算**：R01、R05 这类需求，base 的直觉方案已经足够
2. **单独使用时缺乏质量保障**：dims 没有写测试，纯粹的结构分析不能保证正确性
3. **叠加使用时耗时增加**：dims+super 在 R04 耗时 56 分钟，是 base 的 1.4 倍
4. **对 AI 模型的依赖**：方法论需要模型能够理解并执行分析流程，弱模型可能无法有效利用

### 建议

- 简单需求（改动 < 3 个文件）：不需要方法论
- 中等需求（跨模块变更）：使用变维拆分分析即可
- 复杂需求（跨轮次累积变更）：变维拆分 + Superpowers 组合效果最佳
